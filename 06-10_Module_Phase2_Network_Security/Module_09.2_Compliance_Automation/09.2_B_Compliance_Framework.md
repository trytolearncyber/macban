📘 Module 09.2 — Compliance Framework (Section B)

📌 S — Scenario (Nord Bank / Banking-Specific)

Nord Bank-এর Internal Audit Team বছরে দুইবার Manually প্রতিটা Firewall-এর Rule Check করে দেখত PCI-DSS Requirement মানা হচ্ছে কিনা — যেমন কোনো Unrestricted Rule আছে কিনা, Default Password ব্যবহার হচ্ছে কিনা। এই Manual Review-এ প্রায় ২ সপ্তাহ সময় লাগত, এবং এর মধ্যে যদি কোনো নতুন Non-Compliant Rule যোগ হয়ে যেত, তা পরের Audit পর্যন্ত ধরা পড়ত না।

🚨 Challenge

- Compliance Check বছরে মাত্র দুইবার হতো, তার মাঝে কোনো Non-Compliant পরিবর্তন অনেক দিন ধরা পড়ত না
- Manual Review-এ সময় বেশি লাগত এবং Human Error-এর সুযোগ ছিল (কোনো Rule Miss হয়ে যাওয়া)
- Non-Compliant পাওয়া গেলে সেটা ঠিক করার জন্য আলাদা Manual Process লাগত
- Audit Report তৈরি করতেও Manually অনেক সময় ব্যয় হতো

✅ Solution

একটা Automated Compliance Check তৈরি করতে হবে যা নিয়মিতভাবে Firewall Rule সংগ্রহ করে, PCI-DSS-এর নির্দিষ্ট Requirement-এর বিপরীতে Check করে, এবং Non-Compliant কিছু পাওয়া গেলে সাথে সাথে জানিয়ে দেবে।

🎯 T — Task

Nord Bank-এর জন্য একটা PCI-DSS Compliance Automation Framework ডিজাইন করতে হবে, যা নিয়মিত Firewall Rule Check করবে এবং Non-Compliance ধরা পড়লে Report তৈরি করবে।

👀 O — Output

একটা Compliance Automation Document যেখানে থাকবে:
- Firewall Rule Collection Design — কতক্ষণ পরপর, কোন Device থেকে
- PCI-DSS Requirement Checklist (যেমন Rule Review, Password Policy, Encryption) কীভাবে যাচাই হবে
- Non-Compliance পাওয়া গেলে কী হবে (Alert, Report, নাকি সরাসরি Fix চেষ্টা)
- Compliance Report Generation Design

🤔 R — Reason

বছরে দুইবার Manual Audit যথেষ্ট না, কারণ এর মাঝে অনেক পরিবর্তন হতে পারে যা কেউ লক্ষ্য করে না। নিয়মিত, স্বয়ংক্রিয় Compliance Check করলে Non-Compliant পরিবর্তন দ্রুত ধরা পড়ে এবং সমস্যা ছোট থাকতেই সমাধান করা যায়, যা Regulatory Risk কমায়।

🏦 Real-World Use Case

Nord Bank প্রতি সপ্তাহে একবার সব Firewall-এর Rule সংগ্রহ করে PCI-DSS Checklist-এর বিপরীতে Check করে। কোনো Rule যদি "Any-Any" Access দেয় (কোনো Restriction ছাড়া), সেটা সাথে সাথে Non-Compliant হিসেবে চিহ্নিত হয় এবং Compliance Team-কে Email Report যায়।

🧠 Memory Tip

Compliance Automation-কে ভাবা যেতে পারে একটা "নিয়মিত Health Check-Up"-এর মতো — বছরে একবার Doctor-এর কাছে যাওয়ার বদলে, নিয়মিত Check করলে সমস্যা ছোট থাকতেই ধরা পড়ে।

⚠️ L — Limitation

- Automated Check শুধু আগে থেকে ঠিক করা Rule যাচাই করতে পারে — নতুন ধরনের Compliance Requirement বা জটিল, Context-Dependent Violation এটা নিজে থেকে বুঝতে পারবে না
- একটা Rule Technically Compliant দেখালেও, সেটা আসলে ব্যবসায়িক দিক থেকে ঠিক Configure করা কিনা তা Automation নিশ্চিত করতে পারে না
- Automated Check নিজে থেকে কোনো Non-Compliant Rule "Fix" করতে দিলে (Human Review ছাড়া), ভুল বুঝে সঠিক কোনো Exception Rule-ও মুছে ফেলতে পারে
- Compliance Report তৈরি হলেই Regulatory Audit Pass নিশ্চিত হয় না — Auditor নিজে Manual ভাবে যাচাই করতে চাইতে পারে, বিশেষত Critical System-এ

✋ Y — Your Turn

Nord Bank-এর জন্য PCI-DSS-এর আরেকটা Requirement (যেমন Password Policy বা Encryption) চিন্তা করে লিখতে হবে — Automated Check সেটা কীভাবে যাচাই করতে পারে এবং কোন অংশটা তখনও Human Review দরকার হবে বলে মনে হয়, ২-৩ বাক্যে ব্যাখ্যা করতে হবে।
