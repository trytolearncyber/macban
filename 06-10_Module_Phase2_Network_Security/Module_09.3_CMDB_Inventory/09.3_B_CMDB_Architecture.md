📘 Module 09.3 — CMDB Architecture (Section B)

📌 S — Scenario (Nord Bank / Banking-Specific)

Nord Bank-এর একটা Audit-এর সময় জানতে চাওয়া হয়েছিল ঠিক কতগুলো Network Device Bank-এর Infrastructure-এ আছে এবং কোনগুলো এখনো পুরনো, Unsupported Firmware চালাচ্ছে। কিন্তু কোথাও একটা Central, হালনাগাদ তথ্যভাণ্ডার ছিল না — বিভিন্ন Team-এর কাছে বিভিন্ন সময়ের আলাদা Excel Sheet ছিল, যেগুলো একে অপরের সাথে মিলছিল না। Audit শেষ করতে কয়েক সপ্তাহ লেগে যায় শুধু সঠিক তথ্য একত্র করতেই।

🚨 Challenge

- একটা Central, নির্ভরযোগ্য Device Inventory ছিল না — তথ্য ছড়িয়ে ছিটিয়ে ছিল
- Manually Excel Update করা হতো, যা প্রায়ই পুরনো হয়ে যেত
- নতুন কোনো Device যোগ হলে বা পুরনো কোনো Device সরিয়ে ফেলা হলে, সেটা Inventory-তে সবসময় Reflect হতো না
- Audit বা Security Review-এর সময় সঠিক তথ্য জোগাড় করতে অনেক সময় নষ্ট হতো

✅ Solution

একটা CMDB (Configuration Management Database) তৈরি করতে হবে যা Network-এ থাকা Device-গুলো স্বয়ংক্রিয়ভাবে খুঁজে বের করে (Auto-Discovery), এবং নিয়মিত সেই তথ্য হালনাগাদ রাখে, যাতে সবসময় একটা Central, Up-to-Date Inventory পাওয়া যায়।

🎯 T — Task

Nord Bank-এর জন্য একটা CMDB Architecture ডিজাইন করতে হবে, যা Network Device Auto-Discover করবে, একটা Central Database-এ সংরক্ষণ করবে, এবং নিয়মিত Update রাখবে।

👀 O — Output

একটা CMDB Architecture Document যেখানে থাকবে:
- Auto-Discovery Design — SNMP বা অন্য পদ্ধতিতে কীভাবে Device খুঁজে বের করা হবে
- Data Model — প্রতিটা Device-এর জন্য কী কী তথ্য রাখা হবে (Device Type, Location, Firmware Version, Status)
- Update Mechanism — কতক্ষণ পরপর তথ্য Refresh হবে এবং পরিবর্তন কীভাবে Detect হবে
- Reporting Access — Audit বা Security Team কীভাবে এই তথ্য সহজে দেখতে পারবে

🤔 R — Reason

Manually Maintain করা Inventory সবসময় পুরনো হয়ে যাওয়ার ঝুঁকিতে থাকে, কারণ প্রতিটা পরিবর্তনের পর কাউকে মনে করে Update করতে হয়। Auto-Discovery ব্যবহার করলে Inventory নিজে থেকেই বাস্তব অবস্থা প্রতিফলিত করে, যা Audit, Security Review, এবং দৈনন্দিন Operation-এর জন্য একটা নির্ভরযোগ্য Single Source of Truth তৈরি করে।

🏦 Real-World Use Case

Nord Bank প্রতি রাতে একটা SNMP-ভিত্তিক Scan চালায় যা সব Network Device খুঁজে বের করে এবং তাদের Firmware Version, Status, এবং Location-এর তথ্য একটা Central Database-এ Update করে। পরের বার Audit হলে, Team সরাসরি এই Database থেকে Report বের করতে পারে, Excel খোঁজাখুঁজি করতে হয় না।

🧠 Memory Tip

CMDB-কে ভাবা যেতে পারে একটা "Library-এর Catalog System"-এর মতো — প্রতিটা বই কোথায় আছে তা মনে রাখার বদলে, একটা Central Catalog দেখলেই বলা যায় কোন বই কোথায়, এবং নতুন বই এলে Catalog নিজে থেকেই Update হয়।

⚠️ L — Limitation

- Auto-Discovery সব ধরনের Device সমানভাবে খুঁজে বের করতে পারে না — কিছু Device SNMP Support না করলে বা ভুলভাবে Configure থাকলে Inventory-তে Miss হয়ে যেতে পারে
- CMDB শুধু "কী আছে" তা বলে, "কেন আছে" বা "কে দায়িত্বে আছে" এই ধরনের Context নিজে থেকে বুঝতে পারে না — সেটা আলাদাভাবে Manually যোগ করতে হয়
- একটা ভুল বা Stale Entry যদি দীর্ঘদিন ধরা না পড়ে, তাহলে CMDB নিজেই ভুল তথ্যের উৎস হয়ে যেতে পারে, Excel-এর মতোই
- Network Access না থাকা বা Isolated থাকা Segment-এর Device (Security কারণে) Auto-Discovery থেকে বাদ পড়তে পারে, যা Inventory-কে Incomplete করে দিতে পারে

✋ Y — Your Turn

Nord Bank-এর একটা Device Type চিন্তা করে লিখতে হবে যেটা হয়তো Auto-Discovery দিয়ে সহজে খুঁজে পাওয়া কঠিন হবে (যেমন Isolated Segment-এ থাকা কোনো Device), এবং কেন কঠিন হবে তা ২-৩ বাক্যে ব্যাখ্যা করতে হবে।
