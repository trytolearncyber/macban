📘 Module 08.1 — GitOps for Network Configuration (Section B)

📌 S — Scenario (Nord Bank / Banking-Specific)

Nord Bank-এর একটা Branch-এর Firewall-এ একজন Engineer জরুরি ভিত্তিতে একটা Rule পরিবর্তন করে, কিন্তু কোথাও লিখে রাখেনি ঠিক কী পরিবর্তন করেছে বা কেন করেছে। কয়েক সপ্তাহ পর আরেকজন Engineer লক্ষ্য করে যে Firewall-এর Behavior Documentation-এর সাথে মিলছে না, কিন্তু কেউ বলতে পারে না কখন বা কে এই পরিবর্তন করেছিল।

🚨 Challenge

- Configuration Change-এর কোনো History ছিল না — কে, কখন, কী পরিবর্তন করেছে তা কোথাও Record হয়নি
- Documentation এবং Actual Configuration সময়ের সাথে আলাদা (Drift) হয়ে গিয়েছিল, কিন্তু কেউ ধরতে পারেনি
- একাধিক Engineer আলাদাভাবে Change করলে একে অপরের কাজ Overwrite হওয়ার ঝুঁকি ছিল
- Emergency Change-এর ক্ষেত্রে Approval বা Review ছাড়াই Config পরিবর্তন হয়ে যাচ্ছিল

✅ Solution

প্রতিটা Device-এর Configuration নিয়মিত সংগ্রহ করে একটা Version Control System (Git)-এ সংরক্ষণ করতে হবে, যাতে প্রতিটা পরিবর্তনের একটা History থাকে এবং Current Config বনাম Git-এ থাকা Config-এর মধ্যে পার্থক্য (Drift) স্বয়ংক্রিয়ভাবে ধরা যায়।

🎯 T — Task

Nord Bank-এর জন্য একটা GitOps Architecture ডিজাইন করতে হবে, যেখানে Network Device Configuration নিয়মিত Git Repository-তে Backup হবে এবং Configuration Drift স্বয়ংক্রিয়ভাবে সনাক্ত হবে।

👀 O — Output

একটা GitOps Architecture Document যেখানে থাকবে:
- Scheduled Config Collection Design (কতক্ষণ পরপর, কোন Device থেকে)
- Git Repository Structure — কীভাবে Device অনুযায়ী Config সংগঠিত থাকবে
- Commit History-এর মাধ্যমে Change Tracking (কে না হলেও অন্তত কখন এবং কী পরিবর্তন হয়েছে)
- Drift Detection Logic — Current Config এবং Git-এ থাকা শেষ Config-এর মধ্যে পার্থক্য থাকলে Alert

🤔 R — Reason

Banking Network-এ Configuration Change ট্র্যাক করতে না পারা একটা বড় Audit এবং Security ঝুঁকি — কারণ Regulator বা Internal Audit যেকোনো সময় জানতে চাইতে পারে কখন কী পরিবর্তন হয়েছিল এবং কেন। Git-এর মতো একটা Version Control System ব্যবহার করলে প্রতিটা Change-এর একটা Timeline তৈরি হয়, যা Documentation-এর কাজ করে এবং Drift দ্রুত ধরা সম্ভব করে।

🏦 Real-World Use Case

Nord Bank প্রতিদিন রাতে একটা Automation চালায় যা প্রতিটা FortiGate এবং Cisco Device থেকে Current Configuration সংগ্রহ করে এবং Git Repository-তে নতুন Commit হিসেবে Save করে। পরদিন সেই নতুন Config-কে আগের দিনের Config-এর সাথে তুলনা করা হয় — কোনো অপ্রত্যাশিত পরিবর্তন পাওয়া গেলে Network Team-কে Email পাঠানো হয়।

🧠 Memory Tip

GitOps-কে ভাবা যেতে পারে একটা "Diary"-এর মতো — প্রতিদিন কী ঘটেছে তা লেখা থাকে, তাই পরে ফিরে গিয়ে দেখা যায় ঠিক কবে থেকে কী বদলেছে।

⚠️ L — Limitation

- Git Commit শুধু "কী পরিবর্তন হয়েছে" দেখায়, কিন্তু "কেন পরিবর্তন হয়েছে" তা আলাদাভাবে Commit Message-এ না লিখলে বোঝা যায় না — Automation নিজে থেকে Change-এর কারণ জানে না
- Scheduled Collection-এর মাঝে যদি কেউ একটা Config Change করে আবার সেটা ফিরিয়ে নেয়, তাহলে সেই সাময়িক পরিবর্তনটা কখনো Detect হবে না
- Drift Detection শুধু জানায় যে কিছু বদলেছে, এটা নিজে থেকে বলতে পারে না সেই পরিবর্তনটা ইচ্ছাকৃত (Authorized) নাকি অনিচ্ছাকৃত/অননুমোদিত
- Emergency Change-এর ক্ষেত্রে GitOps Process মেনে Approval নেওয়ার সময় না থাকলে, System-টা এড়িয়ে যাওয়ার Pressure তৈরি হতে পারে, যা পুরো Purpose-কেই ব্যর্থ করে দেয়

✋ Y — Your Turn

একটা পরিস্থিতি চিন্তা করে লিখতে হবে যেখানে Drift Detection একটা পরিবর্তন ধরেছে, কিন্তু সেটা আসলে একজন Authorized Engineer-এর ইচ্ছাকৃত এবং সঠিক পরিবর্তন ছিল — এই ধরনের False Alarm কমানোর জন্য কী করা যেতে পারে তা ২-৩ বাক্যে লিখতে হবে।
